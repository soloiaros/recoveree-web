# Recoveree Web — Coach Dashboard

This document is the single source of truth for the **Coach's Web Dashboard** sub-app
of the Recoveree platform. It is written so a new contributor can pick up work
without reading the rest of the codebase first.

> Sister app: an iOS app (separate repo) that athletes use to log sleep + recovery
> data. This web app **does not** write to `recovery_logs`; it only reads them.
> Athletes are also created in the iOS app — the web app cannot sign athletes up.

---

## 1. Tech Stack

| Concern        | Choice                                |
|----------------|---------------------------------------|
| Framework      | React 18 (function components + hooks)|
| Build tool     | Vite 5                                |
| Backend        | Supabase (Postgres + Auth)            |
| Data client    | `@supabase/supabase-js` v2 (browser)  |
| Styling        | Hand-rolled tokens + CSS in `src/styles.css`, Apple-inspired (system fonts, dark/light themes, SF Symbol-like inline icons, frosted glass, soft shadows). **No Tailwind / MUI / etc.** Replace later if/when a real design system arrives. |
| State / routing| Local component state + a single `AuthContext`. **No React Router.** Tabs are tracked in `useState` inside `DashboardPage`. |

## 2. Quick Start

```bash
npm install
cp .env.example .env   # fill in Supabase URL + anon key
npm run dev            # http://localhost:5173
npm run build          # production sanity check
```

### Required env vars

```
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=...
```

`src/lib/supabaseClient.js` throws at load time if either is missing.
It also **strips any path** from `VITE_SUPABASE_URL` (so pasting the REST
endpoint by mistake doesn't produce an "Invalid path" error). Both legacy JWT
anon keys (`eyJ...`) and the new `sb_publishable_...` keys are supported.

## 3. Database Schema (Supabase, RLS disabled for hackathon)

> Do **not** change these schemas — the iOS app and any seeders depend on them.

### `profiles`
| Column      | Type      | Notes                                                          |
|-------------|-----------|----------------------------------------------------------------|
| id          | uuid (PK) | Matches `auth.users.id`                                        |
| email       | text      | Lowercased on coach signup; whatever the iOS app stored for athletes (we look up case-insensitively) |
| full_name   | text      | Display name; nullable                                          |
| avatar_url  | text      | Public URL into the `avatars` storage bucket; nullable          |
| role        | text      | `'athlete'` or `'coach'`                                       |

### Storage bucket: `avatars`
Public bucket. Filenames are `${userId}-${random}.${ext}`. The web app uploads
new avatars from `useCoachProfile.updateProfile`, then writes the resolved
public URL into `profiles.avatar_url`. Old files are not garbage-collected
(known limitation, see §12).

### `team_roster`
| Column      | Type        | Notes                                   |
|-------------|-------------|-----------------------------------------|
| id          | bigint (PK) |                                         |
| coach_id    | uuid        | `profiles.id` of a coach                |
| athlete_id  | uuid        | `profiles.id` of an athlete             |

### `recovery_logs`
| Column          | Type        | Notes                                |
|-----------------|-------------|--------------------------------------|
| id              | bigint (PK) |                                      |
| athlete_id      | uuid        | `profiles.id`                        |
| sleep_hours     | numeric     |                                      |
| recovery_score  | integer     | 0–100                                |
| ai_advice       | text        | Generated by iOS app / AI pipeline   |
| created_at      | timestamp   |                                      |

> RLS is **disabled** for the hackathon; the anon key has full read/write.
> Policy hints if it's enabled later: a coach should be able to select/insert
> their own `team_roster` rows, select `profiles` whose id is in their roster,
> select `recovery_logs` whose `athlete_id` is in their roster, and upsert
> their own `profiles` row.

> **Future schema work:** the web app currently has no `teams` table — team
> name is stored in `localStorage`. There's also no `is_injured` flag on
> `recovery_logs`; injury status is inferred from `recovery_score` + keywords
> in `ai_advice` (see `src/lib/recoveryStatus.js`).

### Conventions

- **One component per file.** Default export = the component.
- **Hooks** live in `src/hooks/`, prefixed `useXxx`.
- **All Supabase calls go through `src/lib/supabaseClient.js`.** Components
  and hooks import `supabase` from there; nothing else creates a client.
- **Status / formatters are centralized** in `src/lib/recoveryStatus.js` so the
  Overview, Members card, and Detail view all classify the same way.
- **No backend / API routes.** Data is always fetched client-side.

## 5. Auth Flow

`src/context/AuthContext.jsx` + `src/App.jsx`.

1. On mount, `AuthProvider` calls `supabase.auth.getSession()` and subscribes
   to `onAuthStateChange`. `session` is the source of truth.
2. `App.jsx` renders `<AuthPage />` when there's no session, else `<DashboardPage />`.
3. **Sign up:** `supabase.auth.signUp(...)`. Because email verification is
   disabled in the project's Supabase Auth settings, `data.session` is
   returned immediately. We then `upsert` a `profiles` row with
   `id = data.user.id`, `email`, and `role = 'coach'` — this is what makes a
   web signup a coach.
4. **Log in:** `supabase.auth.signInWithPassword(...)`. Profile row is assumed
   to already exist (from signup).
5. Sessions persist across reloads (`persistSession: true`).

> If email verification is re-enabled later, move the profile upsert from
> "right after `signUp`" into the `onAuthStateChange` listener (gated on
> `event === 'SIGNED_IN'` and a missing profile row).

## 6. UI Shell — Header + Pill Nav

`DashboardPage` renders three pieces:

1. **`DashboardHeader`** — frosted-glass bar with brand mark, team name, the
   coach's avatar + display name (from `useCoachProfile`), a theme toggle, and
   a Log out button. Falls back to the auth email if the coach hasn't set a
   `full_name` yet, and to gradient initials if they haven't uploaded an avatar.
2. **`PillNavBar`** — floating, sticky-to-top, frosted-glass segmented control
   with three icon-labeled options: **Overview**, **Members**, **Manage**.
   Implements `position: sticky` + `backdrop-filter: blur(20px)`. Active state
   uses an elevated pill with subtle shadow (matching iOS segmented control
   behavior across light and dark themes).
3. **`<main class="app-main">`** — renders the active tab.

The active tab lives in a single `useState('overview')` inside `DashboardPage`.
If we ever need shareable URLs per tab, swap that state for
`window.location.hash` or React Router.

## 7. Overview Tab

`src/components/overview/OverviewTab.jsx`.

- Greeting line based on local time.
- **Three stat cards** powered by `summarizeTeam(athletes)` from
  `recoveryStatus.js`:
  - **Need recovery** (orange) — score 50–74
  - **Injury recovery** (red) — score < 50, or `ai_advice` contains an
    injury keyword (`injury`, `strain`, `pain`, etc.)
  - **Fully rested** (green) — score ≥ 75
- Each card shows the count, an inline progress bar (count / total), and a
  percentage. Athletes with no logs yet are bucketed as "unknown" and don't
  count toward any of the three.
- **Recent activity** card listing the 6 most recent logs across the team
  (uses `allLogs` from `useTeamData`).
- A **Refresh** button re-runs the team data query.

---

## 8. Members Tab

`src/components/members/MembersTab.jsx`.

- Renders a responsive grid of `<AthleteCard>` components: `<Avatar />` (image
  from `profiles.avatar_url` or gradient initials fallback), display name
  (from `profiles.full_name`, falling back to email), status badge, latest
  score, and "last log Xh ago". The athlete's email is shown as a subtitle
  when it differs from `full_name`.
- Clicking a card stores the athlete's id in local state and swaps the grid
  for `<AthleteDetail>`. A "← Members" button returns to the grid.

`AthleteDetail`:
- Header with the athlete avatar (image or initials), display name, email
  subtitle, status badge, and last-updated timestamp.
- Three KV cards (recovery score / sleep hours / total logs on file).
- A highlighted card showing the latest `ai_advice` string.
- `LogTimeline` rendering the 8 most recent logs, vertical timeline style.

> The detail view filters `allLogs` (already loaded by `useTeamData`) by
> `athlete_id` rather than refetching, which keeps the interaction snappy.

## 9. Manage Tab

`src/components/manage/ManageTab.jsx`.

- **`ProfileEditor`** — coach can edit their own `full_name` and upload an
  avatar image. Backed by `useCoachProfile`. The flow is exactly the
  reference snippet documented in §13.1: optionally upload the file to the
  public `avatars` Storage bucket, resolve a public URL via
  `getPublicUrl`, then `update` the `profiles` row with the new `full_name`
  and (if a file was uploaded) `avatar_url`. The new avatar shows in the
  header and across the dashboard immediately because `useCoachProfile`
  updates its local state from the `update().select()` round-trip.
- **`TeamNameEditor`** — input + Save button, persisted via `useTeamName`
  (localStorage, scoped per coach id). Used by the header.
- **`InviteAthleteForm`** — coach enters the athlete's email. We look the
  email up case-insensitively in `profiles` (`ilike`), reject coach profiles
  and roster duplicates, and only insert into `team_roster` once we've
  confirmed the athlete's profile id exists — so the FK on `team_roster.athlete_id`
  can never trigger a raw constraint error in the UI. If the email isn't in
  `profiles`, we tell the coach the athlete needs to sign up in the iOS app first.
  Calls `onAdded` to refresh the dashboard.
- **`RosterManagementList`** — current roster with a "Remove" button per row.
  Removal asks for `window.confirm()`, then deletes the matching
  `team_roster` row (does **not** touch `recovery_logs`).

## 10. Status Classifier + Identity Helpers (`lib/recoveryStatus.js`)

Single source of truth. Exports:
- `STATUS` — enum of `{ injured, needsRecovery, rested, unknown }`, each with
  `id`, `label`, `short`, `color`, `icon`.
- `classifyAthlete(athlete)` — returns one `STATUS` value based on
  `latestLog.recovery_score` + injury-keyword scan over `ai_advice`.
- `summarizeTeam(athletes)` — counts per bucket; used by the Overview cards.
- `displayName(profile, fallback?)` — preferred name string (`full_name` →
  email → fallback). Used by every UI surface that needs to label a person.
- `personInitials(profile)` — two-letter initials, derived from `full_name`
  if set, otherwise from email. Used by the gradient `<Avatar />` fallback.
- `formatDateTime`, `formatRelative`, `emailToInitials` — UI helpers.

When the schema gets a real injury flag, edit `classifyAthlete` and nothing
else needs to change.

## 11. Common Tasks

**Add a new tab:**
1. Add an entry to `TABS` in `DashboardPage.jsx`.
2. Add a new tab component under `src/components/<your-tab>/`.
3. Add a branch to the `activeTab === '...'` block in `DashboardPage`.

**Show a new field from `recovery_logs`** (e.g. `hrv`):
1. Add it to the `select(...)` lists in `useTeamData.js`.
2. Render it in `AthleteCard`, `AthleteDetail`, and/or `RecentActivityList`.

**Show a new field from `profiles`** (e.g. `phone_number`):
1. Add it to the `select(...)` in `useTeamData.js` (for athletes) and
   `useCoachProfile.js` (for the current coach).
2. Spread the new field into the merged athlete object in `useTeamData`.
3. Render it wherever needed; if it's identity-related, consider extending
   `displayName` / `personInitials` instead of duplicating logic.

**Persist team name to Supabase** (when a `teams` table exists):
- Replace `useTeamName` with a Supabase-backed version. Keep the same return
  shape `{ teamName, setTeamName }` so callers don't change.

**Switch to URL-based routing:**
- Replace the `useState` in `DashboardPage` with React Router. The pill nav
  becomes `<NavLink>` items styled the same way.

## 12. Known Limitations / Tech Debt

- No tests.
- No realtime subscriptions; coach pulls latest data via the Refresh button.
- Team name stored in `localStorage` (no `teams` table yet).
- Injury status inferred (no `is_injured` column on `recovery_logs`).
- No email-based invite flow — the athlete must already exist in `profiles`
  (created in the iOS app).
- Avatar uploads use a random suffix per save, so old files accumulate in the
  `avatars` Storage bucket forever. Clean up periodically or switch to a
  deterministic name + cache-busting query string.
- Errors are surfaced as plain strings; no toast system.
- RLS is off — anyone with the anon key can read all data.

### 13.1 Avatar upload reference flow

The exact pattern implemented in `useCoachProfile.updateProfile`:

```js
// 1. Optionally upload to the public `avatars` bucket
if (file) {
  const ext = file.name.split('.').pop();
  const fileName = `${userId}-${Math.random()}.${ext}`;
  const { error } = await supabase.storage.from('avatars').upload(fileName, file);
  if (!error) {
    const { data } = supabase.storage.from('avatars').getPublicUrl(fileName);
    avatarUrl = data.publicUrl;
  }
}

// 2. Update the profile row
const updateData = { full_name: fullName };
if (avatarUrl) updateData.avatar_url = avatarUrl;
await supabase.from('profiles').update(updateData).eq('id', userId);
```

The hook hardens that snippet by surfacing upload/update errors instead of
swallowing them and by re-selecting the row so callers see the new state
without a manual refetch.
